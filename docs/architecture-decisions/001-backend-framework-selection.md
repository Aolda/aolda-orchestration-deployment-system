# ADR 001: Backend API Framework Selection

## Date
2026-03-31

## Context (배경)
내부 앱 배포 플랫폼의 API 백엔드를 구축함에 있어 초기에 Fiber 프레임워크가 제안되었다. 그러나 향후 시스템을 유지 관할 주체가 전업 백엔드 개발자가 아닌 **DevOps 엔지니어**이며, 대부분의 코드를 **AI (Codex 등)** 의 도움을 받아 생성 및 관리하려는 제약 사항(Context)이 존재했다.

## Decision (의사 결정)
*외부 서드파티 프레임워크(Fiber, Gin 등)를 전면 배제하고, **Go 1.22+ 버전에 포함된 표준 라이브러리(`net/http`)**를 라우터와 핸들러 기반 스펙으로 절대 채택한다.* 

또한, 폴더 구조는 AI의 컨텍스트 파악을 돕기 위해 복잡한 Layered 구조를 버리고 도메인 단위(`internal/application`, `internal/project`)의 직관적 분할 방식(Flat Domain-driven)을 도입한다.

## Rationale (선정 및 판단 기준 - 포트폴리오용)
1. **AI 코딩 정확도 극대화**: AI 에이전트(LLM)는 세상에서 가장 많은 오픈소스 코드를 학습한다. Go 언어의 경우 압도적으로 표준 라이브러리 기반의 HTTP 서버 풀이 넓다. 특정 프레임워크의 문법이나 버전에 따른 할루시네이션(잘못된 메서드 호출)을 완전히 차단할 수 있다.
2. **Go 1.22 패치 보완**: 1.22 버전부터 `net/http` 기본 라우터가 강력해져 `GET /api/v1/projects/{id}` 와 같은 Path 변수 맵핑이 서드파티 라우터 모듈 없이도 네이티브로 충실하게 지원된다.
3. **유지보수 비용(TCO) 최소화**: DevOps 운영자 입장에서 서드파티 프레임워크의 생태계 변화나 업그레이드 여파(Breaking Changes)를 신경 쓸 필요 없이 인프라 안정성에 집중할 수 있다.
