# ADR 002: Frontend UI & CSS Framework Selection

## Date
2026-03-31

## Context (배경)
내부 플랫폼의 대시보드 및 복잡한 앱 생성 폼(YAML 대신 설정 정보를 입력받는 뷰)을 구축해야 한다. 초기 기획에서는 최근 유행하는 Tailwind CSS 스택을 고려했으나, **전담 디자이너가 배치되지 않은 인하우스 프로젝트**이며, React 코드 대부분을 에이전트(AI)가 뽑아내야 하고 유지보수 주체 역시 DevOps 엔지니어라는 현실적 제약이 있었다.

## Decision (의사 결정)
*Tailwind CSS 정책을 폐기하고, UI 컴포넌트 렌더링에는 **Mantine (v7)** 라이브러리를 핵심 스펙으로 규정한다. 추가적인 커스텀 스타일링이 필요할 때는 **Vanilla CSS (CSS Modules)** 방식을 엄격하게 적용한다.*

## Rationale (선정 및 판단 기준 - 포트폴리오용)
1. **Tailwind의 제한성과 AI 컨텍스트 소모**: 수십 개의 직관적이지 않은 유틸리티 클래스가 덕지덕지 붙어 있는 코드는 프론트 전문 인력이 아니면 구조를 한눈에 파악하기 매우 어렵다. 또한, AI 모델이 이 클래스들을 생성하느라 토큰 창구를 무의미하게 소모하여 로직 구현 컨텍스트를 떨어뜨리는 원인이 된다.
2. **사전 제작된 모던 UI의 압도적 퀄리티 (No Designer)**: Mantine은 Material UI(무거움, 올드함)나 Ant Design(커스텀의 어려움)의 대표적 단점들을 극복한 현대적인 어드민 대시보드용 컴포넌트(모달, 폼 바인딩, 데이터 테이블)를 기본 제공한다. 디자이너가 없는 상태에서도 프로덕션 수준의 세련된 UI를 보장받을 수 있다.
3. **순수 CSS 기반의 우아함 (CSS Modules 네이티브 호환)**: 최신 Mantine v7 아키텍처는 CSS Modules 위에 설계되었다. 즉, AI가 로직을 짤 때 추가적인 설정 파일 튜닝이나 충돌 걱정 없이 단순히 `import classes from 'A.module.css'`로 깔끔하게 CSS 분리 구조를 구성할 수 있으므로, 유지보수 측면의 가시성이 극대화된다.
