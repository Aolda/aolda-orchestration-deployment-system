AODS SETTINGS GUIDE

저장소 연결, 이미지 Pull, 모노레포 처리까지 한 번에 보는 설정 페이지

새 애플리케이션 생성 화면에서 입력하는 값들을 어디서 만들고 왜 필요한지 정리한 페이지입니다. GitHub 저장소 토큰, GHCR 이미지 Pull 토큰, aolda_deploy.json 형식, 프론트와 백엔드가 같은 저장소일 때의 처리 방식, 그리고 이미지 태그 운영 원칙까지 여기에서 바로 확인할 수 있습니다.

GitHub 저장소 토큰 발급 GHCR 토큰 발급

바로 가기

  • 입력 순서
  • 입력 조합 빠른 판단
  • 저장소 설정
  • 이미지 / 레지스트리 설정
  • 프론트+백엔드 같은 레포
  • aolda_deploy.json 형식
  • AODS 저장 경로
  • 이미지 태그 운영 원칙

입력 순서

처음 연결할 때는 아래 순서로 진행하면 됩니다. 제일 먼저 이 순서만 보고 따라가면 됩니다.

  1. GitHub 저장소 안에 aolda_deploy.json을 만든다.
  2. 서비스가 여러 개면 serviceId를 구분해서 넣는다.
  3. 이미지를 private registry에 올린다면 레지스트리 사용자명과 토큰을 준비한다.
  4. AODS에서 GitHub 저장소 URL을 넣는다.
  5. 필요하면 repositoryServiceId를 넣는다.
  6. private 저장소면 저장소 읽기 토큰을 넣는다.
  7. private 이미지면 레지스트리 사용자명과 레지스트리 토큰을 넣는다.
  8. 배포 후에는 새 이미지를 만들 때마다 새 tag를 발급하고 descriptor도 같이 갱신한다.

입력 조합 빠른 판단

내 저장소와 이미지가 public인지 private인지에 따라 어떤 값을 넣어야 하는지 먼저 판단하면 제일 쉽습니다.

1. public 저장소 + public 이미지

가장 단순한 조합입니다.

  • GitHub 저장소 URL만 넣으면 됩니다.
  • 저장소 토큰: 비워 둠
  • 레지스트리 사용자명 / 토큰: 비워 둠
2. private 저장소 + public 이미지

설정 파일을 읽으려면 저장소 읽기 권한이 필요합니다.

  • GitHub 저장소 URL
  • GitHub 저장소 토큰
  • 레지스트리 사용자명 / 토큰: 비워 둠
3. public 저장소 + private 이미지

설정 파일은 읽히지만 이미지를 pull 할 권한이 따로 필요합니다.

  • GitHub 저장소 URL
  • 저장소 토큰: 선택
  • 레지스트리 사용자명 + 레지스트리 토큰
4. private 저장소 + private 이미지

두 종류의 인증이 모두 필요합니다.

  • GitHub 저장소 URL
  • GitHub 저장소 토큰
  • 레지스트리 사용자명 + 레지스트리 토큰

저장소 설정

AODS가 GitHub에서 실제로 읽는 값은 저장소 URL, 브랜치, 설정 파일 경로, 그리고 필요한 경우 저장소 읽기 토큰입니다.

GitHub 저장소 URL
  • 예: https://github.com/aods/example-app.git
  • 기본 브랜치를 읽는 구조지만, 필요하면 브랜치를 따로 지정할 수 있습니다.
  • public 저장소면 URL만으로도 시작할 수 있습니다.
GitHub 저장소 토큰
  • private 저장소이거나 GitHub API rate limit을 피하고 싶을 때 넣습니다.
  • 권장 종류: Fine-grained personal access token
  • 필요 권한: Repository permissions → Contents → Read-only
입력 예시
  • repositoryUrl: https://github.com/jalju0804/moltbot-front-poc.git
  • repositoryBranch: main
  • configPath: aolda_deploy.json
  • repositoryServiceId: 저장소 안의 serviceId와 정확히 같아야 합니다.

이미지 / 레지스트리 설정

저장소 토큰과 이미지 Pull 토큰은 목적이 다릅니다. 저장소를 읽는 권한과 이미지를 내려받는 권한은 분리해서 관리합니다.

레지스트리 주소
  • 보통 ghcr.io, docker.io, public.ecr.aws 같은 값을 넣습니다.
  • 이미지 주소에서 추론 가능하면 비워도 되지만, private 레지스트리는 명시하는 편이 안전합니다.
레지스트리 사용자명 / 토큰
  • private 이미지일 때만 필요합니다.
  • GHCR 기준 사용자명은 이미지에 접근 가능한 GitHub 계정 ID입니다.
  • GHCR 기준 최소 권한은 read:packages 입니다.
항목 의미 예시
registryServer 이미지 pull 대상 레지스트리 주소 ghcr.io
registryUsername 레지스트리에 접근 가능한 사용자명 jalju0804
registryToken 이미지 pull 용 token 또는 password GHCR classic PAT
중요
  • 저장소 토큰은 aolda_deploy.json을 읽기 위한 값입니다.
  • 레지스트리 토큰은 Kubernetes가 컨테이너 이미지를 pull 하기 위한 값입니다.
  • 둘은 같은 값일 수도 있지만, AODS에서는 분리해서 저장하는 것을 기본으로 봅니다.

프론트와 백엔드가 같은 레포일 때

현재 AODS는 저장소 단위가 아니라 서비스 단위로 앱을 나눕니다. 같은 레포라도 프론트와 백을 각각 별도 앱으로 등록하는 방식이 맞습니다.

권장 방식
  • 같은 저장소에 web, api 두 서비스를 둡니다.
  • aolda_deploy.json의 services[]에 둘 다 적습니다.
  • AODS에서는 앱을 2개 등록합니다.
  • 둘 다 같은 repositoryUrl을 쓰고, repositoryServiceId만 다르게 넣습니다.
왜 이렇게 하나
  • 프론트와 백엔드는 보통 이미지, 포트, replicas, rollout 전략이 다릅니다.
  • 운영 관점에서도 배포/롤백 단위를 분리하는 편이 안전합니다.
  • secret, registry credential, 이력도 앱 단위로 분리하는 편이 관리가 쉽습니다.
{
  "services": [
    {
      "serviceId": "moltbot-web",
      "image": "ghcr.io/org/moltbot-web:sha-abc1234",
      "port": 3000,
      "replicas": 1,
      "strategy": "Rollout"
    },
    {
      "serviceId": "moltbot-api",
      "image": "ghcr.io/org/moltbot-api:sha-abc1234",
      "port": 8080,
      "replicas": 1,
      "strategy": "Rollout"
    }
  ]
}
현재 제한
  • AODS는 지금 frontend/backend를 자동으로 나눠 인식하지 않습니다.
  • 같은 레포에 있더라도 서비스별 serviceId를 명시해야 합니다.
  • 즉, 저장소 1개를 연결하더라도 앱은 서비스별로 각각 등록한다고 생각하면 됩니다.

aolda_deploy.json 형식

AODS는 이 파일에서 이미지, 포트, replicas, strategy를 읽습니다. 서비스가 하나면 하나만, 여러 개면 services 배열에 계속 추가합니다.

필수에 가까운 값
  • serviceId: AODS에서 찾을 서비스 식별자
  • image: 배포할 컨테이너 이미지
  • port: 서비스 포트
  • replicas: 기본 레플리카 수
선택값
  • strategy: Rollout 또는 Canary
  • 전략을 안 넣으면 팀 운영 규칙에 맞춰 기본 전략을 따르게 구성하는 편이 좋습니다.
{
  "services": [
    {
      "serviceId": "example-app",
      "image": "ghcr.io/aods/example-app:sha-abc1234",
      "port": 3000,
      "replicas": 1,
      "strategy": "Rollout"
    }
  ]
}

AODS 저장 경로

AODS는 앱 환경변수, 저장소 토큰, 이미지 Pull 토큰을 서로 다른 경로에 분리해서 저장합니다.

종류 경로 설명
앱 환경변수 secret/aods/apps/{projectId}/{appName}/prod 애플리케이션 내부 환경변수와 비밀값
저장소 토큰 secret/aods/apps/{projectId}/{appName}/repository GitHub 저장소 설정 파일을 읽기 위한 토큰
레지스트리 인증 secret/aods/apps/{projectId}/{appName}/registry Kubernetes image pull credential
현재 모델
  • secret은 현재 앱 단위로 나뉩니다.
  • 환경별로 완전히 다른 secret tree를 자동 분리하는 구조는 아직 기본 계약이 아닙니다.
  • 따라서 프론트와 백엔드가 같은 저장소라도 앱은 분리해서 등록하는 편이 더 현실적입니다.

이미지 태그 운영 원칙

AODS는 현재 descriptor 안의 이미지 tag 변화를 기준으로 새 배포를 판단합니다. 그래서 immutable tag 운영이 훨씬 잘 맞습니다.

권장
  • sha-1a2b3c4d
  • v2026.04.17-001
  • GitHub Actions run number 기반 tag
비권장
  • latest 같은 mutable tag 덮어쓰기
  • 같은 tag를 registry에 다시 push 하는 방식
왜 same tag overwrite를 피하나
  • AODS는 현재 tag 문자열이 바뀌었는지 중심으로 새 배포를 감지합니다.
  • 같은 tag를 덮어써도 Git의 값이 안 바뀌면 AODS는 새 배포로 보지 않을 수 있습니다.
  • 운영 추적, 롤백, 이력 관리도 새 tag를 매번 발급하는 방식이 훨씬 명확합니다.
