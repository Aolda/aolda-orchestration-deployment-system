2026-04-18: LoadBalancer exposure workflow must not be inferred from Flux sync alone. Added backend GET /api/v1/applications/{applicationId}/network-exposure that reads Kubernetes Service.spec.type, Service.status.loadBalancer.ingress, and latest Service Event. Response statuses: Internal, Pending, Provisioning, Ready, Error. Frontend App.tsx loadBalancerExposureWorkflow now consumes this endpoint so stage 3 can show real provisioning/error/ready state and stage 4 only becomes active after ingress address exists. Local mode still returns Pending with explanatory message when loadBalancerEnabled=true but Kubernetes API is not configured. Related files: backend/internal/kubernetes/real.go, backend/internal/kubernetes/local.go, backend/internal/application/service.go, backend/internal/server/server.go, frontend/src/App.tsx, docs/internal-platform/openapi.yaml.
